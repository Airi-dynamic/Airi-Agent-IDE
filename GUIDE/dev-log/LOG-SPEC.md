# 开发日志撰写规范（LOG-SPEC）

> 本文是 `dev-log/` 目录下所有 `dayN-*.md` 文件的强制规范。  
> 撰写或修订任意一份日志前，必须先阅读本规范，并保证产出符合本文每一条要求。  
> 当本规范与既有日志发生冲突时，**以本规范为准，旧日志应被改写**。

---

## 0. 设计哲学

### 0.1 一句话定位
> **日志 = 一份读者可以照着从零敲键盘的迭代式教程，而不是一份解释成品的事后说明书。**

### 0.2 三个根本原则

1. **过程式（Process-Oriented），不是结果式（Result-Oriented）。**
   - 反例：「我们最终的 `chat.service.ts` 长这样：[贴 200 行]」+「下面解释每段代码的作用」。
   - 正例：「先建空类与构造函数 → 加最小 `sendMessage` 桩 → 接入 Provider → 加 abort 信号 → 改造成异步流」。读者每完成一步都能保存、运行、观察现象。
2. **教程顺序，不是文件顺序，更不是特性顺序。**
   - 步骤排列必须满足「**依赖在前，使用在后**」：当读者敲入第 N 步代码时，第 N 步引用的所有符号、文件、模块都必须已经在前面 1..N-1 步中创建好。
   - 严禁让读者写一个组件却引用一个尚未创建的 store。
3. **完整、不省略。**
   - 每一个**新增文件**必须展示其**完整最终内容**（不允许 `// ...` 省略号）。
   - 每一个**修改文件**必须以**完整新内容**或**精确 diff 块**给出，且让读者能确切定位到改了哪几行。严禁只贴「关键片段」而把其余隐式留给读者推断。
   - 但凡当日仓库（`GUIDE/dayN/` 与 `GUIDE/dayN-1/` 的差异集合）里出现的代码改动，**必须在日志中至少出现一次**。

### 0.3 三条反例（一旦出现即视为不合规）

| 反例形式 | 为什么不行 |
|---|---|
| 「核心增量一：XXX 服务」开头直接贴完整文件，再分段解释 | 这是结果式说明，不是过程教程 |
| 用「`...其余略`」「`// 同 Day N 不变`」省略代码 | 违反完整性原则 |
| 章节按文件分组（4.1 Service / 4.2 IPC / 4.3 Preload / 4.4 Store / 4.5 Component），但文件之间存在依赖且未按依赖顺序排列 | 违反教程顺序原则 |

## 1. 文件命名

### 1.1 文件名格式
```
dayN-<topic-summary>.md
```

- `N`：阿拉伯数字，如 `1`、`12`、`28`，**不要补零**。
- `<topic-summary>`：3~12 个汉字或英中混排，简明概括当日主题。允许中文、连字符、字母数字；禁止空格、下划线、扩展名外的标点。
- 旧的 `README-dayN.md` 命名**全部弃用**。

### 1.2 命名示例
- `day1-基础设施搭建.md`
- `day2-编辑器与终端.md`
- `day6-Agent主循环.md`
- `day20-MCP客户端.md`

## 2. 必备章节结构

每份日志严格按以下编号顺序排布。除非显式标记为「按需」，所有章节必须存在。

```
# Day N：<完整标题>

## 0. 前言
## 1. 本日目标与产出
## 2. 先跑起来（Smoke Run）
## 3. 项目结构与变更总览
   ### 3.1 项目目录树（ASCII）
   ### 3.2 构建 / 运行链路图
   ### 3.3 编码步骤索引（依赖顺序）
## 4. 基础知识铺垫（Day 1/2 必写；后续按需，仅讲当日新引入的概念）
## 5. 编码步骤（核心，按依赖顺序的迭代教程）
## 6. 端到端串联走查（一条用户操作的事件链路）
## 7. 完整运行流程追踪（结合真实示例的编译/启动/运行时流程）
## 8. 必学知识点深化（按需）
## 9. 自测清单
```

下面逐节给出强制要求。

> **范式定位**：日志读者应被假设为「会写一点 JS、但对 Node 项目结构、TypeScript、Electron 与 Agent 工程都不熟」的开发者。Day 1/2 必须把当日全部预备知识在 § 4 一次性补齐；Day 3 起，仅当日新引入的概念才在 § 4 出现。

## 3. 各章节强制要求

### 3.1 § 0 前言
- 一段话回答：**前一日交付了什么？今日要补什么缺口？为什么必须现在补？**
- 不超过 200 字，不含代码。

### 3.2 § 1 本日目标与产出
- 用编号列表分别给出 **功能目标** 与 **工程目标**。
- 每一条目标必须可在 § 7 自测清单中找到对应验证项。

### 3.3 § 2 先跑起来
- 必须先告诉读者**跑通当日已完成版本**的方式（`Set-Location .\GUIDE\dayN; npm install; npm run dev`）。
- 必须给出**预期现象**（截图描述、控制台关键日志、UI 文案）。
- 这一节让读者在动手之前先看到目标形态，建立"我要做出什么"的心象。

### 3.4 § 3 项目结构与变更总览

这一节由三个固定子节构成，缺一不可。

#### 3.4.1 § 3.1 项目目录树（ASCII）
- 给出**当日完成后** `GUIDE/dayN/` 的完整目录 ASCII 树（用 `├─ └─ │` 字符）。
- 对每个**新增**或**变化**的文件 / 目录，行尾用 `# 新增` 或 `# 修改` 标注。
- 目的：让读者一眼建立"今天到底做了什么文件级变更"的空间感。

#### 3.4.2 § 3.2 构建 / 运行链路图
- 用一段 ```text 代码块画出从 `npm run dev`（或当日相关入口命令）到首屏 / 首条响应之间的调用链。
- 链路图的每一跳标注**实际涉及的文件与符号**。
- 这一节回答：「我跑命令以后，电脑里到底依次发生了什么？」

#### 3.4.3 § 3.3 编码步骤索引（依赖顺序）
- 给出一份「编码步骤索引」**表格**，格式固定为四列：

| # | 文件 | 类型 | 说明 |
|---|---|---|---|
| 1 | `package.json` | 新增 / 修改 | 说明本步骤做了什么 |
| 2 | `electron/services/git.service.ts` | 新增 | simple-git 封装 |
| … | … | … | … |

- **禁止**使用 ` ```text ` 代码块替代表格。
- 「类型」列只允许填 `新增` 或 `修改`。
- 「说明」列用一句话点明本步骤新增/修改的核心内容，不超过 20 字。
- 排列规则：**底层依赖（类型、纯函数、服务）在前；顶层视图与组合层在后。**
- 步骤编号必须与 § 5 内的二级标题编号一一对应。

### 3.4bis § 4 基础知识铺垫

- **Day 1 / Day 2 必写**；Day 3 起，**仅当日新引入的概念**（库、协议、算法、范式）才在此节出现。
- 撰写视角：假设读者**第一次**接触本节话题，先讲「这是什么 / 为什么需要 / 在本项目哪里出现」，再贴最小代码示例（不必是当日代码原文，可用合成示例）。
- 推荐子节模板（按当日实际涉及挑选）：
  - 4.X **Node.js / npm / package.json 是什么**（仅 Day 1）
  - 4.X **TypeScript 入门**：类型注解 / 接口 / 泛型 / 模块 / JSX / 声明文件 / 严格模式（仅 Day 1）
  - 4.X **Electron 三进程模型**：main / preload / renderer 与 contextIsolation（仅 Day 1）
  - 4.X **当日新增外部库速览**：每个库 3~6 行说明（库做什么、本日用到哪个 API、可替代品）
  - 4.X **Agent 应用整体结构与本日的位置**（Day 1 必写一次；Day 4/5/6 各加一节深化）
  - 4.X **新概念深讲**：如 Day 4 的 `AsyncIterable`、Day 5 的 JSON Schema、Day 8 的 LCS
- 篇幅指导：每个新概念至少 80 字描述 + 一个最小可读代码块；**不要堆术语清单**。
- 完成本节后，读者进入 § 5 时应当不再需要查外部资料即可理解每段代码。

### 3.5 § 5 编码步骤（核心）

这是日志的核心章节，每一个 § 3.3 列出的步骤都必须是 § 5 的一个二级标题：

```
### 5.1 步骤 1：在 package.json 增加 simple-git 依赖
### 5.2 步骤 2：建立 GitService 骨架……
…
```

每个二级标题下，必须依次包含以下五个固定小节（即使内容很短也要保留）：

#### A. 动机
- 1~3 句话，回答「**为什么这一步要这么做？为什么是现在做？**」。
- 严禁只讲"我们要新增 XXX 文件"，必须有意图层面的解释。

#### B. 你将要做的操作
- 用一句祈使句告知读者具体动作：
  - 「在 `electron/services/` 下新建文件 `git.service.ts`，写入下面完整内容。」
  - 「打开 `electron/preload.ts`，在 `contextBridge.exposeInMainWorld('api', {...})` 内追加以下方法。」

#### C. 完整代码（强制完整）

- **新增文件**：必须给出完整文件内容，禁止省略。
- **修改文件**：以下两种格式二选一，**必须能让读者一字不差地完成修改**：
  - **格式 A（替换块）**：先标注「在文件 X 第 N 行附近，将原代码替换为：」再给完整新片段。
  - **格式 B（完整新版）**：当修改占文件 70% 以上时，直接重贴整文件。
- 严禁出现 `// ...其余不变`、`// 同上`、`// 略`。如果某段确实保持不变，要么贴出来，要么用 § 3.5C 「上下文锚点」格式：
  ```ts
  // 上下文锚点 ↓ 以下三行保持不变，仅用于定位
  function foo() { ... 已存在 ... }
  // 上下文锚点 ↑

  // ↓↓↓ 新增 ↓↓↓
  function bar() { ... }
  // ↑↑↑ 新增 ↑↑↑
  ```

#### D. 这段代码做了什么（解释）
- **强制要求**：必须做到「**逐段、按上下文顺序、把每一处对读者陌生的语法和行为都讲清楚**」，覆盖：
  1. **每一行 / 每一段做了什么**（行为层面的解释，不能只说"配置了 XXX"）。
  2. **为什么这么写**（设计意图）。
  3. **如果改成另一种常见写法会怎么样**（对比性反例，至少出现 1 次/步，可省的步骤需明确说明"无可对比写法"）。
- 当某段代码涉及 § 4 没覆盖的细微语法（如非空断言 `!`、可选链 `?.`、`as const` 等），就地补 1~3 行解释；不要假设读者"应该已经懂"。
- 紧扣本步骤新增 / 修改的代码做解释，**不要**重复讲解之前步骤已经存在的内容。
- 解释顺序应当与代码上下顺序一致。

#### E. 立刻验证
- 给出**这一步完成后**就可以做的最小验证：
  - 「保存后 `npm run dev`，在 DevTools Console 输入 `await window.api.gitStatus()`，应当返回包含 `current` 字段的对象。」
  - 或：「保存后查看 TypeScript 编译输出，应当无新增错误。」
- 如果某一步必须等到后续步骤才能验证，明确写出：「本步骤无法独立验证，将在步骤 X 之后统一运行。」并解释原因。

### 3.6 § 6 端到端串联走查
- 在所有步骤完成后，用 **一条具体用户操作**（例如「在 Chat 中说『帮我读 package.json』」、「点击侧栏 ✕ 折叠」）作为线索，按真实调用链顺序追踪一次：
  - 渲染层事件 → preload → IPC → service → provider → 工具 → 回调 → IPC 推送 → store → UI 重渲染。
- 每个跳转点贴出**实际经过的关键代码行**（带文件名），形成全栈追踪表。
- 与 § 7 区别：§ 6 聚焦 **运行时事件链**，回答「用户某次操作发生了什么」。

### 3.7 § 7 完整运行流程追踪（结合真实示例）
- 与 § 6 互补：§ 7 聚焦**进程 / 编译 / 加载层面的整体流程**，回答「我执行 `npm run dev` 之后，机器内部按什么顺序做了哪些事，最终为什么能弹出窗口、为什么我的代码会被运行到」。
- 必须**结合真实文件与代码片段**：从命令行入口（`scripts.dev`）开始，依次贴出被实际触发的配置 / 入口文件 / 模块加载顺序，并解释每一跳的产物与去向。
- 推荐覆盖维度：
  1. 命令行 → 构建器 → 三端编译产物路径
  2. 主进程冷启动 → 创建窗口 → 加载 renderer URL
  3. preload 上下文挂载 → renderer JS 入口 → React 挂载 → 首屏可见
  4. 当日新引入的运行时回路（如 IPC、流式、tool loop）的真实首次激活时序
- 这一节是初学者建立"系统级心智模型"的关键，**Day 1 必写**；后续每一日如有新增运行时链路也要补一次。

### 3.8 § 8 必学知识点深化（按需）
- 仅当 § 4 介绍过的某个概念在当日代码中以**进阶形态**出现，需要专门展开时使用。
- 若当日 § 4 已经讲透、§ 8 无可补，则整节可省略，但需在 § 0 末尾注明。

### 3.9 § 9 自测清单
- 给出可勾选格式：
  ```markdown
  - [ ] 启动后能看到 ⎇ 图标
  - [ ] 点击 ⎇ 切换到 GitPanel，显示当前分支
  - [ ] 修改任意文件后刷新，文件出现在 modified 列表
  ```
- 每条自测项必须**直接对应** § 1 的某条目标。

## 4. 代码块书写规范

### 4.1 代码块语言标记
- 所有代码块必须带语言标识：```ts / ```tsx / ```json / ```powershell / ```text。

### 4.2 文件路径标注
- 在每个代码块之上 1 行用斜体或行内代码注明路径：
  ```
  *文件 `electron/services/git.service.ts`：*
  ```ts
  ...完整代码...
  ```
  ```

### 4.3 修改类代码块的差异表达
- 推荐使用 § 3.5C 中的「上下文锚点」格式。
- 不允许使用 GitHub diff 语法 (`+ -`)，因为读者难以复制；必须给出**直接可粘贴**的最终代码。

### 4.4 行号引用
- 行号会随版本变动失效，**严禁**写「修改第 42 行」。改为「在 `loadSettings()` 方法末尾追加……」这种语义锚点。

## 5. 与 `GUIDE/dayN/` 快照的关系

- `GUIDE/dayN/` 是**只读的最终代码快照**，用于读者校对和回退。
- `dev-log/dayN-*.md` 是**生产那份快照的过程**，二者必须强一致：
  - 日志给出的所有「最终完整代码」必须与 `GUIDE/dayN/` 中对应文件**字节级一致**（除允许的注释差异外）。
  - 撰写日志前的强制检查：`diff -r GUIDE/day(N-1) GUIDE/dayN` 列出的每一个变化项，都必须能在日志的 § 3 步骤索引中找到归属。

- [ ] § 0~§ 9 全部章节齐备，编号正确。
- [ ] § 3.1 ASCII 目录树存在并标注新增 / 修改。
- [ ] § 3.2 构建链路图存在并标注真实文件 / 符号。
- [ ] § 3.3 步骤索引按依赖顺序排列。
- [ ] § 4 已对当日新引入的概念全部铺垫（Day 1/2 必写；后续按需）。
- [ ] § 5 每一步都包含动机 / 操作 / 完整代码 / 详细行为解释 / 立刻验证五段。
- [ ] § 5 D 段做到了「逐段讲行为」+「至少一次对比反例」。
- [ ] 任意一段代码都没有 `// ...` 或「略」。
- [ ] `GUIDE/dayN/` 与 `GUIDE/day(N-1)/` 的完整 diff 已被本日志 100% 覆盖。
- [ ] § 6 端到端走查贴的代码片段确实存在于 § 5 已经写过的代码中。
- [ ] § 7 完整运行流程追踪结合了真实文件与命令。
- [ ] § 9 每条自测项都能在 § 1 找到对应目标。

## 7. 版本演进

- 本规范本身随 day 推进可能调整。任何修改必须：
  1. 在文件末尾以「## 变更记录」追加条目，写明 day 序号、修改人、原因。
  2. 同步审视既有日志是否仍然符合新规范。
